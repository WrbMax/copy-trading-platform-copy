# Route 智能跟单平台 - 详细产品说明文档 (v1.8)

## 1. 产品概述

Route 智能跟单平台是一款基于 AI 信号驱动的加密货币自动化交易系统。用户通过注册账户、充值 USDT 作为平台服务费余额，并绑定第三方交易所（如 Binance、OKX 等）的 API Key，即可一键订阅平台提供的量化交易策略。

当平台接收到来自策略提供方的 Webhook 信号时，系统会自动在所有订阅该策略的用户的真实交易所账户中执行开平仓操作。平台采用“**先盈利、后付费**”的模式：只有当跟单产生的交易实际盈利平仓后，系统才会从用户的平台余额中扣除 40% 的净盈利作为服务费。这 40% 的服务费中，一半（20%）用于多级市场推广分润，另一半（20%）进入平台流动性池。

## 2. 核心业务流程

### 2.1 账户与资金流转
* **注册与邀请机制**：平台实行强邀请制，新用户注册必须填写有效的邀请码。注册成功后，用户自动生成自己的专属邀请码。
* **专属充值地址**：平台使用 HD 钱包技术为每位用户在 BSC（BNB Smart Chain）网络上生成独一无二的 USDT 充值地址。
* **自动化充值检测**：系统每 3 分钟自动扫描一次链上数据（结合 BSCScan API 与 RPC 节点余额快照双重检测防重），发现用户向专属地址转入 USDT 后，自动将等额资金记入用户的平台余额。
* **提现审核**：用户可申请提取平台余额至任意 BSC 链地址。提现申请需管理员在后台人工审核，审核通过后系统自动通过主钱包完成链上转账。

### 2.2 策略与跟单执行
* **API 绑定**：用户在平台输入交易所的 API Key、Secret Key（及 Passphrase），系统使用 AES-256-GCM 算法加密存储。
* **策略订阅**：用户选择平台提供的策略并设置“跟单倍数”（如 0.1x 到 100x）。
* **余额门槛**：用户的平台余额必须大于 0 才能订阅策略或保持策略运行。
* **信号处理与执行**：
  1. 策略端通过 Webhook 向平台发送交易信号（包含币对、方向、数量、价格等）。
  2. 平台跟单引擎（Copy Engine）接收信号后，遍历所有订阅该策略且余额 > 0 的用户。
  3. 系统根据用户设置的跟单倍数计算实际下单数量（实际数量 = 信号数量 × 跟单倍数），并调用对应交易所的 API 执行下单。
  4. 订单状态在系统中实时记录（Pending -> Open -> Closed / Failed）。

### 2.3 服务费扣除与账户熔断
* **扣费时机**：只有在订单**平仓且产生净盈利**（Net PnL = Realized PnL - 交易所手续费 > 0）时，系统才会触发扣费。
* **扣费比例**：扣除净盈利的 **40%**。
* **熔断机制**：系统从用户的平台余额中直接扣除服务费。如果用户余额不足以支付全额服务费，系统会扣除全部剩余余额（余额清零），并**立即自动暂停**该用户的所有跟单策略，直到用户再次充值。

## 3. 收益分成与市场推广体系

平台扣除的 40% 服务费中，20% 进入平台流动性池，另外 20% 作为"分润池"（Revenue Pool）用于奖励推广者。

**重要说明：分润池是按每一笔订单单独计算和即时分配的，而非累计后统一结算。** 每当一笔跟单订单平仓且产生净盈利时，系统立即以该笔订单的净盈亏（`netPnl`）为基数，计算出本笔分润池（`netPnl × 20%`），并在同一事务中实时分配给推荐链上的所有符合条件的推广者。例如，某用户某笔订单净盈利 100 USDT，则本笔分润池 = 20 USDT，系统立即将这 20 USDT 按规则拆分打入各推荐人余额，与其他订单互不干扰。

分润池的分配遵循严格的数学模型，确保在任何情况下总拨出率不会超过 100%（即净盈利的 20%）。

### 3.1 有效用户定义
* **有效用户**：平台余额 ≥ 100 USDT 的用户即为有效用户。
* **作用**：有效用户数量仅用于计算其直接推荐人（上级）是否满足“分享奖”的解锁门槛。

### 3.2 奖励类型一：分享奖（直推奖）
分享奖仅发放给交易用户的**直接推荐人**（上一级）。奖励比例取决于该推荐人名下拥有的“有效直推用户”数量。

| 有效直推人数 | 分享奖比例（占分润池） |
| :--- | :--- |
| < 3 人 | 0% |
| 3 - 5 人 | 8% |
| 6 - 8 人 | 10% |
| ≥ 9 人 | 15% |

### 3.3 奖励类型二：身份奖（级别奖 - 极差制）
身份奖发放给交易用户上方整个推荐链条上的"P 身份"达标者。

**P 身份考核规则（平衡腿模式）：**
P 身份的评定不再基于简单的总伞下业绩，而是采用**“去大区，取小区”**的考核标准。
1. **业绩定义**：伞下用户盈利订单扣除的服务费总额（即净盈利的 40%）。
2. **大区剔除**：系统计算用户所有直推分支的整棵子树业绩，将其中业绩最大的一个分支视为“大区”并剔除。
3. **小区考核**：剩余所有分支的业绩之和即为“小区业绩”，当小区业绩达到下表门槛时，即可获得对应 P 身份。
*(注：如果用户只有一个直推分支，则该分支即为大区，小区业绩为 0，无法升级 P 身份。)*

| 身份级别 | 升级门槛（小区业绩累计） | 身份奖比例（占分润池） |
| :--- | :--- | :--- |
| P1 | 1,000 USDT | 10% |
| P2 | 3,000 USDT | 20% |
| P3 | 10,000 USDT | 30% |
| P4 | 30,000 USDT | 35% |
| P5 | 100,000 USDT | 40% |
| P6 | 300,000 USDT | 45% |
| P7 | 1,000,000 USDT | 55% |

**极差分配逻辑（发放时大区业绩正常参与计算）**：
系统从下往上遍历推荐链，计算每个祖先节点的 P 身份。如果当前祖先的身份比例高于已被截留的比例，则该祖先获得两者的“差额”。
*例如：用户 A 某笔订单净盈利 500 USDT，产生本笔分润池 100 USDT（500 × 20%）。其直接上级 B 是 P1（10%），再上级 C 是 P3（30%）。则 B 获得 100 × 10% = 10 USDT；C 获得 100 × (30% - 10%) = 20 USDT；剩余 70 USDT 留在分润池中未被分配（因为链条上没有更高 P 身份的节点）。*

### 3.4 奖励类型三：平级奖（同级拦截补偿）
当推荐链上出现“下级与上级 P 身份相同”时（例如 P3 推 P3），由于极差为 0，上级原本无法获得身份奖。为补偿上级，系统设立了平级奖。
* **触发条件**：链条上某节点与其直属下级（在推荐链中紧挨着的下一级）拥有相同的 P 身份，且该 P 身份 > 0。
* **奖励金额**：分润池的 **10%**。
* **限制**：一笔订单的分润过程中，平级奖**最多只向上触发一次**。

## 4. Route 积分生态系统

为弥补用户在跟单过程中的亏损，平台引入了 Route 积分作为补偿和生态流转凭证。

* **获取方式（自动空投）**：当用户的跟单订单平仓并产生净亏损时，系统会按照 `1 USDT 净亏损 = 1 Route` 的比例，实时、自动向用户账户发放 Route 积分（向下取整）。
* **流转机制**：Route 积分可以在平台用户之间自由转让。转让时只需输入对方的平台邀请码即可实时到账。
* **生态价值**：Route 积分可用于平台后续的生态权益兑换（具体兑换场景由平台后续公布）。

## 5. 前端功能模块拆解

### 5.1 用户端 (Client)
1. **首页落地页 (Landing)**：展示平台核心特性、实时交易数据滚动播报、注册引导。
2. **仪表盘 (Dashboard)**：核心数据看板。展示平台余额、Route 积分、累计盈利/亏损、净盈亏曲线图，以及最近 5 笔交易记录。
3. **策略中心 (Strategy)**：展示平台提供的所有 AI 策略。用户可在此绑定 API、设置跟单倍数并一键开启/关闭策略。
4. **订单记录 (Orders)**：完整展示用户的所有跟单记录，包括币对、方向、数量（ETH）、开平仓价格、时间和净盈亏。支持分页和状态筛选。
5. **我的收益 (Earnings)**：展示用户作为推广者获得的收益。包含分享奖、身份奖、平级奖的实时流水列表，以及累计收到和被扣除的分润统计。
6. **系统收益 (Team)**：推广者的团队管理中心。展示当前 P 身份、大区/小区业绩详情、分享有效人数进度（如 `3 / 6 人`）、系统总人数和总业绩。提供直推成员列表，可下钻查看每个成员的具体交易流水。
7. **充值提现 (Funds)**：获取专属 USDT 充值地址（二维码），提交提现申请，以及查看所有资金变动流水（充值、提现、服务费扣除、分润入账）。
8. **Route 中心 (Points)**：查看 Route 积分余额，进行积分转让操作，以及查看积分获取（亏损补偿）和转让的明细流水。
9. **API 绑定 (ExchangeApi)**：管理各交易所的 API 密钥，支持一键连通性测试。

### 5.2 管理后台 (Admin)
1. **数据大盘 (Dashboard)**：全站核心数据总览。包括系统总用户数、平台净收入（40% 中的 20%）、多级分润总支出（40% 中的 20%）、流动性池余额、全站累计交易额与盈亏。
2. **用户管理 (Users)**：查看所有注册用户详情。支持封禁/解封用户、手动调整用户余额（如线下入金）、强制修改用户的 P 身份级别。
3. **资金管理 (Funds)**：
   * **充值审核**：查看系统自动检测的充值记录。
   * **提现审核**：审批用户的提现申请。拒绝时系统会自动将资金（标记为 `admin_adjust`）退回用户平台余额。
   * **钱包状态**：监控平台 HD 主钱包的 BNB（用于 Gas）和 USDT 余额。
4. **订单监控 (Orders)**：全局查看所有用户的跟单执行情况。重点监控异常/失败订单（如交易所 API 报错、余额不足等），并支持手动将订单标记为异常。
5. **收益分成 (Revenue Share)**：全局视角的推广分润账单。统计平台累计发放的直推奖、级别奖和平级奖总额，并提供所有分润记录的明细流水。
6. **Route 管理 (Points)**：全局 Route 积分流水监控。管理员可在此为任意用户手动增加或扣除 Route 积分。
7. **信号源管理 (Signals)**：配置和管理平台策略。生成 Webhook Secret 供外部策略系统对接调用。

## 6. 数据库核心模型 (Schema)

* **users**: 存储用户基础信息、平台余额 (`balance`)、Route积分 (`points`)、P身份 (`pLevel`) 和伞下业绩 (`umbrellaPerformance`)。
* **exchange_apis**: 存储用户绑定的交易所 API 密钥（AES加密）。
* **signal_sources**: 平台策略配置及 Webhook 密钥。
* **copy_orders**: 核心跟单记录。记录开平仓价格、数量、`netPnl`（净盈亏）以及 `revenueShareDeducted`（已扣除的服务费）。
* **revenue_share_records**: 分润流水表。记录每一笔分润的来源订单、来源用户、收款人、奖励类型（`direct`/`rank`/`same_rank`）、比例和金额。
* **fund_transactions**: 资金变动总账本。记录充值、提现、分润收入、服务费支出等所有导致 `balance` 变动的行为。
* **points_transactions**: Route 积分账本。记录亏损发放、转账、后台调整等行为。
* **deposits / withdrawals**: 充提币业务表，记录链上 TxHash 和审核状态。

## v1.9 (Security & Auto-Payout Update)

### 1. 提现自动打币功能 (Auto-Payout)
- **规则调整**：
  - ≤ 200 USDT 的提现申请：系统自动审核通过并立即发起链上转账。
  - > 200 USDT 的提现申请：需管理员在后台手动审核通过，通过后系统自动发起链上转账。
- **主钱包余额挂起机制**：当主钱包的 USDT 或 BNB（Gas）余额不足时，提现申请状态将保持为 `pending` 挂起，不会拒绝或报错。
- **自动重试机制**：系统后台每 5 分钟运行一次定时任务，扫描所有 `pending` 状态的提现，并在主钱包余额充足时自动完成打币。

### 2. 资金安全与幂等性修复 (Idempotency)
- **提现防重复打币（高危修复）**：引入 `claimWithdrawalForProcessing` 原子锁机制，通过数据库级别的 `UPDATE ... WHERE status='pending'` 确保定时任务和管理员并发审核时，同一笔提现绝对不会被执行两次链上转账。
- **充值防重复到账（中危修复）**：
  - RPC 余额检测方式生成的合成 `txHash` 改为基于 `地址 + 金额` 的确定性键值，确保同一笔充值无论扫描多少次都能被正确去重。
  - 充值余额更新逻辑从非原子的 `SELECT + UPDATE` 升级为基于 `adjustBalance` 的原子 SQL 操作，彻底杜绝并发扫描导致的丢失更新或重复加款问题。
